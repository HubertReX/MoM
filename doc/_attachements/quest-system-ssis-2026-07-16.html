1 · Graf zależności questów (DAG)

DAG (Directed Acyclic Graph — skierowany graf acykliczny) to graf zależności: węzły to questy, a strzałki pokazują kierunek odblokowywania. Uwaga: nazwa quests_hierarchy w kodzie jest myląca — zależności krzyżują się między głównymi questami, więc to graf, nie drzewo.

Strzałki ciągłe to łańcuch unlocks (ukończenie A odblokowuje B), strzałki przerywane to relacja "należy do" z quests_hierarchy (główny → jego podquesty). Węzły główne są wypełnione kolorem akcentu, podquesty mają tylko obwódkę. Najedź myszą, by zobaczyć szczegóły questa, lub kliknij, by otworzyć panel poniżej.

Idiosynkrazje. Kolejność pól w JSON jest niespójna (stare questy zaczynają od bonus, nowe Q* od name). unlocks bywa false, null lub stringiem. bonus bywa 1-kluczowy (legacy) lub 6-kluczowy (rodzina Q*) — dlatego logika break na pierwszej niezerowej wartości ma znaczenie.

Mapowanie QUEST_BONUSES_MESSAGES (z utils.py). Iterowane w tej kolejności — pierwsza niezerowa wygrywa, reszta jest ignorowana (nawet jeśli inne są niezerowe).

Jak działa aplikacja (quests.py:50)

def apply_quest_bonus(quest, cfg):
    bonuses = quest["bonus"]
    hero = cfg["hero"]
    value = 0
    for bonus_key in QUEST_BONUSES_MESSAGES.keys():  # money→health→max_health→hp→agility→eloquence→max_items
        value = bonuses.get(bonus_key, 0)
        if value != 0:
            if bonus_key != "max_items":
                setattr(hero, bonus_key, getattr(hero, bonus_key) + value)
            else:
                hero.inventory.max_items += value   # specjalny przypadek: torba
            break                                    # ← tylko jedna nagroda na quest
    hero.health = max(0, min(hero.health, hero.max_health))  # clamp zdrowia
    return value
Konsekwencja portowania. Jeśli quest ma {"agility":5,"eloquence":5}, gracz dostanie TYLKO +5 zwinności. max_items leży na hero.inventory, nie na hero — w innej grze musisz wskazać odpowiednią lokację atrybutu. Clamp health dzieje się zawsze, nawet przy nagrodzie niezwiązanej ze zdrowiem.

Zarówno test, jak i test_progress są eval-owane jako Python z całym cfg w scope (funkcje check_quest i check_quest_progress).

Co jest w scope eval-a

Zmienne
  • hero — postać gracza (atrybuty, inventory.items, unique_loc_visited_cnt, searched_loc, coc_for/until/cmd)
  • characters["klucz"] — is_met, hp, dialog.key
  • curr_loc — name_key, is_searched
  • coc — łańcuch komend (replay)
  • len — dodane jawnie
Blokady bezpieczeństwa
  • cfg["__builtins__"] = {}
  • cfg["__import__"] = {}
  • skutek: brak import, brak wbudowanych funkcji
Ryzyko. Scope to cały cfg — test ma dostęp do całego stanu gry. W nowej grze pomyśl o jawnym, węższym słowniku kontekstu zamiast eval(globals=cfg).

Po pętli po zwykłych questach, check_quests przechodzi quests_hierarchy. Główne questy mają dwa tryby ukończenia:

Kind 1 · Agregacja

test == "False" / False

Główny quest kończy się, gdy wszystkie jego podquesty mają is_done. Liczy je count_done_quests.

Przykłady: 02_game_mechanics, 03_speed_run, Q01_S00_BREAK_THE_CURSE, Q02_S00_TRINKETS_FOR_SARCASMIA, Q03_S00_LEARN_ABOUT_CURSE.

Kind 2 · Self-test

test = prawdziwe wyrażenie

Główny quest kończy się przez własny test (obsłużony już w pierwszej pętli). Pętla hierarchii to tylko sieć bezpieczeństwa (idempotentny mark_quest_done).

Przykłady: 01_main_01 (characters["troll_001"].is_met), Q00_S00_WHAT_IS_GOING_ON (puste podquesty).

for main_quest_key, sub_quest_keys in cfg["quests_hierarchy"].items():

DAG, nie drzewo. Q01_S05_..._MEET_MADAME_SARCASMIA ma unlocks: "Q02_S00_TRINKETS_FOR_SARCASMIA" — podquest z jednego łańcucha odblokowuje główny quest z INNEGO łańcucha. Międzyłańcuchowe unlocks działają od razu, zanim główny quest zostanie sfinalizowany. Autor modelował fabułę jako graf skierowany, mimo nazwy "hierarchy".

check_quests_done() odpala się w dwóch miejscach w main.py:

  • main.py:2022 — w pętli gry, po find_items(), przed check_level_up() (co tick)
  • main.py:2784 — po zakończeniu dialogu (testy oparte na dialog.key zapalają się natychmiast)

Podwójna nagroda. Pętla w main.py nie rozróżnia głównych od podquestów — apply_quest_bonus aplikuje się do każdego zrobionego questa. Gdy główny quest kończy się agregacją, jego własny bonus (np. 01_main_01: +50 max_health) dodaje się do bonusów wszystkich podquestów.

4 · Pułapki przy portowaniu do innej gry

1. eval na całym cfg. Testy mają dostęp do całego stanu gry i są blokowane tylko płytko (__builtins__={}). W nowej grze użyj jawnego, wąskiego kontekstu lub dedykowanego DSL zamiast eval.
2. Tylko pierwsza niezerowa nagroda. apply_quest_bonus robi break na pierwszym niezerowym kluczu (kolejność: money→health→max_health→hp→agility→eloquence→max_items). Reszta bonusów w słowniku jest cicho ignorowana.
3. test_total działa tylko z test_progress. Mianownik paska jest czytany z JSON wyłącznie gdy test_progress jest ustawione i nie jest "False". W przeciwnym razie UI trzyma domyślne 1.
4. Niespójne typowanie unlocks. Występuje jako false (bool), null i string. check_quest chroni się przez if unlocks_key:, ale mark_quest_done robi lookup bez ochrony — tam unlocks musi być stringiem lub null.
5. Graf to DAG, nie drzewo. Międzyłańcuchowe unlocks (np. Q01_S05 → Q02_S00) sprawiają, że zależności krzyżują się między głównymi questami. Planując port, nie zakładaj czystej hierarchii.
6. max_items poza hero. Trzymane na hero.inventory.max_items i obsługiwane osobnym if. W innej grze wskaż właściwą lokację.
7. Literówka w kluczu. Q03_SO3_HOW_TO_GET_THERE — w quests_hierarchy i quests figuruje jako SO3 (powinno być S03). Działa tylko dlatego, że obie strony używają tej samej literówki. Przy porcie ujednolić.
8. Clamp zdrowia zawsze. Po aplikacji nagrody hero.health jest zawsze ucinane do [0, max_health], niezależnie od tego, czy nagroda dotyczyła zdrowia.
